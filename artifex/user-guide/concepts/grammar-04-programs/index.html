    4. Program Distributions and Execution Semantics¤

    Chapter 4 prerequisites and learning objectives¤

    This chapter assumes latent-variable models and basic programming-language ideas. By the end, you should be able to:

    1. O1 — Define a program law: specify a normalized distribution over finite abstract-syntax trees and its domain-specific language.
    2. O2 — Push mass through execution: compute an induced output probability by summing or integrating over programs and execution randomness.
    3. O3 — Separate validity layers: distinguish syntax, static semantics, dynamic safety, functional behavior, and scientific adequacy.
    4. O4 — Combine priors, likelihoods, and search: identify which component defines the target law and which only proposes or verifies candidates.
    5. O5 — Audit termination and equivalence: distinguish construction from execution termination and aggregate mass over declared behavioral classes.
    6. O6 — Transfer the program contract: design a resource-accounted program generator whose posterior and evaluation object are unambiguous.

    A grammar over programs¤

    Let a typed domain-specific language (DSL) provide primitives and production rules. Sampling a derivation produces an abstract syntax tree \(P\):

    \[ P\sim p_\theta(P). \]

    Execution maps the syntax tree and an input or random seed \(u\) to an output:

    \[ y=\operatorname{Exec}(P,u). \]

    If execution is stochastic with \(u\sim p(u\mid P)\), the induced observation law is

    \[ \boxed{ p(y)=\sum_P p_\theta(P) \int p(u\mid P) \mathbf 1\!\left\{\operatorname{Exec}(P,u)=y\right\}\,du. } \]

    For continuous outputs, the indicator is replaced by the appropriate density or pushforward measure. This sum is usually harder than parsing a string because execution can be many-to-one, discontinuous, expensive, or partial.

    Syntax, static semantics, and behavior¤

    Program validity has layers:

    Table GRAM.4. Syntax, static semantics, and behavior
    Layer Example question Typical mechanism
    lexical/syntactic does the token sequence parse? CFG or parser
    static semantics are variables scoped and types compatible? attributes, types, symbol table
    dynamic safety does execution avoid exceptions and resource violations? interpreter, sandbox, abstract interpretation
    functional behavior does it satisfy examples or a specification? tests, theorem prover, constraint solver
    scientific adequacy is the program a plausible explanation? posterior, predictive checks, domain review

    A context-free grammar usually guarantees only the first layer. Attribute or type-directed generation can enforce some static constraints. No finite syntax mask proves total functional correctness for an unrestricted programming language.

    Complete worked example: many programs, one result¤

    Consider the expression DSL

    \[ E\to\operatorname{add}(E,E)\quad(0.4), \qquad E\to 1\quad(0.6). \]

    Execution returns the integer sum of all leaves. The program 1 returns one with probability \(0.6\). The only program returning two is add(1,1), with probability

    \[ 0.4(0.6)^2=0.144. \]

    Two syntax trees return three:

    add(1, add(1, 1))
add(add(1, 1), 1)

    Each has probability \(0.4^2 0.6^3=0.03456\), so

    \[ p(Y=3)=2(0.03456)=0.06912. \]

    The posterior over programs given \(Y=3\) is uniform over these two trees. If the DSL later adds identities such as add(E,0) or multiplication by one, many more semantically equivalent programs may support the same result.

    Priors, likelihoods, and Bayesian program learning¤

    A common posterior has the form

    \[ p(P\mid D)\propto p(P)\,p(D\mid P). \]

    The prior can favor short descriptions, reusable library components, typed productions, or domain-specific motifs. The likelihood may be deterministic agreement with examples, a noisy observation model, or a differentiable renderer. Shorter programs are not automatically truer; description length depends on the chosen language and library.

    Human-level concept learning through probabilistic program induction and memoised wake--sleep illustrate different domains and inference systems. Their empirical results are not universal sample-efficiency guarantees for arbitrary DSLs.

    Search, amortization, and library learning¤

    Exact enumeration is possible only in small, strongly bounded DSLs. Practical systems combine enumerative search, beam search, SMC, MCMC, constraint solving, neural proposals, or cached candidate sets. A recognition model \(q_\phi(P\mid D)\) can prioritize promising programs without becoming the prior \(p(P)\) itself.

    Library learning adds reusable abstractions and therefore changes the language, prior, and program lengths. Bayesian program learning by decompiling amortized knowledge reports an ICML 2024 DreamCoder extension that uses a learned search policy to propose library components. Comparisons must include search budget, timeout, DSL, library-update schedule, and the number of solutions retained.

    Termination and resource-bounded semantics¤

    There are two termination questions:

    1. does the grammar finish constructing a finite program? and
    2. does the constructed program finish executing?

    A subcritical construction grammar can still generate a looping program. A strongly normalizing DSL can guarantee execution termination while its recursive program generator remains inconsistent. Timeouts convert execution into a resource-bounded partial observation map; timeout must be recorded as an outcome or explicitly conditioned away.

    Program equivalence and posterior multiplicity¤

    Many programs can compute the same function. Commutativity, dead code, variable renaming, refactoring, and library identities create equivalence classes. In a general expressive language, exact semantic equivalence is not a tractable generic deduplication operation. Bounded finite DSLs can admit canonical forms or exhaustive behavioral signatures, but those are additional assumptions.

    Posterior mass over functions is

    \[ p(f\mid D)=\sum_{P:\llbracket P\rrbracket=f}p(P\mid D), \]

    not the probability of one shortest representative. Evaluation must therefore say whether diversity is measured over strings, syntax trees, canonical forms, or behavior.

    Independent practice and assessment map: program distributions¤

    Attempt each task before opening its hints. The typical time is a planning aid, not a speed target. Use the success criterion to decide whether to revise before reading the complete solution.

    1. P1.1 — Predict before computing: an output mass. Type: predict · Difficulty: core · Typical time: 10 min. In the expression DSL \(E\to\operatorname{add}(E,E)\) with probability \(0.4\) and \(E\to1\) with probability \(0.6\), predict whether \(P(Y=4)\) exceeds \(P(Y=3)\). Then compute the number of programs returning four and their total mass.
    2. P1.2 — Aggregate equivalent programs. Type: derive · Difficulty: foundation · Typical time: 7 min. For the two equal-mass programs returning three in the worked example, compute \(p(P\mid Y=3)\) and state why reporting either one program's prior mass is not the output probability.
    3. P1.3 — Classify validity failures. Type: diagnose · Difficulty: foundation · Typical time: 9 min. Classify each outcome at its earliest failed layer: an unparsable token sequence; a parsed expression adding an integer to a function; a well-typed program that times out; and a terminating program that fails a held-out specification test.
    4. P1.4 — Combine a prior and likelihood. Type: derive · Difficulty: core · Typical time: 9 min. Two programs have prior probabilities \((0.6,0.4)\) and data likelihoods \((0.2,0.9)\). Compute the normalized posterior and identify which program a prior-only search and a posterior search would prefer.
    5. P1.5 — Keep timeout in the denominator. Type: evaluate · Difficulty: core · Typical time: 10 min. Of 1,000 attempted programs, 700 pass the specification, 200 terminate but fail, and 100 time out. Compute unconditional success, completion rate, and success conditional on completion. Explain which number may be reported as system success.
    6. P1.6 — Transfer. Type: design · Difficulty: stretch · Typical time: 20 min. Define a Bayesian symbolic-regression system with a typed DSL. Specify the program prior, noisy data likelihood, proposal/search role, resource bound, equivalence convention, posterior query, and extrapolation evidence.

    Complete solutions to P1
    1. P1.1. Four leaves have \(C_3=5\) ordered full-binary program shapes. Each has mass \((0.4)^3(0.6)^4=0.0082944\), so \(P(Y=4)=5(0.0082944)=0.041472\). This is less than \(P(Y=3)=0.06912\): multiplicity grows, but the additional recursive and terminal factors reduce total mass here. Check. The answer sums programs, not integer partitions.
    2. P1.2. Each compatible program has posterior probability \(0.03456/0.06912=1/2\). One program's mass \(0.03456\) is only one latent explanation; the output probability sums both to \(0.06912\). Check. The posterior normalizer is precisely the observation mass in this finite case.
    3. P1.3. The token sequence fails lexical/syntactic validity; integer-plus-function fails static type semantics; timeout fails dynamic/resource-bounded execution; and the terminating wrong answer fails functional behavior. Check. Passing an earlier layer is necessary but does not imply passage of a later one.
    4. P1.4. The unnormalized weights are \((0.12,0.36)\), so the posterior is \((0.25,0.75)\). A prior-only ranking prefers the first program; a posterior ranking prefers the second. Check. A proposal trained to find the second program still needs a correction before it represents these posterior probabilities.
    5. P1.5. Unconditional success is \(700/1000=0.70\); completion is \(900/1000=0.90\); and success conditional on completion is \(700/900=7/9\approx0.778\). Overall system success is \(0.70\) when every attempt is in scope. The conditional number may also be reported, but only with its denominator and the timeout rate. Check. Dropping timeouts would inflate the headline result.
    6. P1.6. One valid system uses a typed expression grammar with a depth-aware normalized prior, a Gaussian residual likelihood with declared noise scale, and a neural proposal plus enumerative or SMC search that targets rather than defines the posterior. Timeouts and numerical exceptions are recorded as outcomes under a fixed resource budget. Algebraic canonicalization aggregates identities such as commutative reordering; finite test agreement is labeled a weaker behavioral signature. The query may be posterior predictive mass or posterior program families, and extrapolation is measured on held-out input ranges with calibration and complexity stratification. Rubric. Full credit specifies all seven requested components and keeps proposal quality distinct from posterior correctness.

    Chapter 4 failure cases and misconception checks¤

    • Executable is not correct. Passing a parser and interpreter is weaker than satisfying a specification.
    • Timeout is not rejection for free. Dropping timed-out samples changes the reported distribution.
    • A neural proposal is not the posterior. Importance weights or another correction are needed for posterior claims.
    • Program length is language dependent. A learned macro can make the same behavior appear shorter without changing it.
    Retrieval question: what are the two sources of multiplicity?

    A yield can have several syntactic derivations, and several distinct executable programs can have the same behavior. Both require summing mass when the query concerns the observation or function rather than one trace.

    Chapter 4 synthesis¤

    Program generators place probability on syntax and induce a second law through execution. Types, resource limits, and specifications refine the observation map but remain separate contracts. Chapter 5 composes these structured laws with neural latents, autoregressive and diffusion samplers, circuits, and GFlowNets.
