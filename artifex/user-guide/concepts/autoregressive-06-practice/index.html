    Practical Implementation in Artifex¤

    Practical Implementation in Artifex: Prerequisites and learning objectives¤

    This chapter assumes basic JAX array semantics and the Flax NNX object model. Examples explicitly distinguish current Artifex APIs, executable NNX reference implementations, and shape-oriented pseudocode. By the end, you should be able to:

    1. construct the three documented Artifex autoregressive model classes;
    2. distinguish live Artifex API examples, executable NNX reference code, and shape-oriented pseudocode;
    3. design shift, mask, and nucleus-boundary tests for a generation implementation;
    4. report likelihood, quality, and systems metrics with their denominators and protocols; and
    5. produce a hardware-aware inference accounting table without invented throughput constants.

    API surface versus algorithmic contract

    The following blocks document current public constructors and method signatures. The executable curriculum contracts separately run the NNX recurrent example, target gathering, and a shifted mock decoder. Before relying on a production sampler, test that logit \(t-1\) produces token \(t\), suffix changes cannot alter earlier logits, top-\(p\) retains its boundary token, and prompt/EOS behavior is correct. These semantic invariants are stronger than “the method returned an array.”

    Basic Autoregressive Model¤

    Live Artifex API example. Variables such as sequences are supplied by the surrounding training pipeline.

    from flax import nnx

from artifex.generative_models.core.configuration import (
    TransformerConfig,
    TransformerNetworkConfig,
)
from artifex.generative_models.models.autoregressive import (
    TransformerAutoregressiveModel,
)

rngs = nnx.Rngs(0)

config = TransformerConfig(
    name="text-transformer-ar",
    vocab_size=10000,
    sequence_length=512,
    num_layers=6,
    dropout_rate=0.1,
    network=TransformerNetworkConfig(
        name="text-transformer-network",
        hidden_dims=(512,),
        activation="gelu",
        embed_dim=512,
        num_heads=8,
        mlp_ratio=4.0,
    ),
)

# Create Transformer autoregressive model
model = TransformerAutoregressiveModel(config, rngs=rngs)

# Training
batch = {"sequences": sequences}  # [batch_size, seq_len]
outputs = model(batch["sequences"], rngs=rngs, training=True)
loss_dict = model.loss_fn(batch, outputs, rngs=rngs)

# Generation
samples = model.generate(
    n_samples=10,
    max_length=256,
    temperature=0.8,
    top_p=0.9,
    rngs=rngs
)

    PixelCNN for Images¤

    Live Artifex API example. Confirm the dataset's integer range, channel ordering, and any quantization or dequantization convention before comparing BPD.

    from flax import nnx

from artifex.generative_models.core.configuration import PixelCNNConfig
from artifex.generative_models.models.autoregressive import PixelCNN

rngs = nnx.Rngs(0)

config = PixelCNNConfig(
    name="mnist-pixelcnn",
    image_shape=(28, 28, 1),
    num_layers=7,
    hidden_channels=128,
    num_residual_blocks=5,
)

# Create PixelCNN for MNIST (28×28 grayscale)
model = PixelCNN(config, rngs=rngs)

# Training
batch = {"images": images}  # [batch_size, 28, 28, 1], values in [0, 255]
outputs = model(batch["images"], rngs=rngs, training=True)
loss_dict = model.loss_fn(batch, outputs, rngs=rngs)

# Generation
generated_images = model.generate(
    n_samples=16,
    temperature=1.0,
    rngs=rngs
)

    WaveNet for Audio¤

    Live Artifex API example. The waveform must use the discrete symbolization declared by vocab_size; raw floating-point PCM is not interchangeable with 8-bit categorical targets.

    from flax import nnx

from artifex.generative_models.core.configuration import WaveNetConfig
from artifex.generative_models.models.autoregressive import WaveNet

rngs = nnx.Rngs(0)

config = WaveNetConfig(
    name="audio-wavenet",
    vocab_size=256,
    sequence_length=16000,
    residual_channels=128,
    skip_channels=512,
    num_blocks=3,
    layers_per_block=10,
)

# Create WaveNet for audio
model = WaveNet(config, rngs=rngs)

# Training
batch = {"waveform": waveform}  # [batch_size, time_steps]
outputs = model(batch["waveform"], rngs=rngs)
loss_dict = model.loss_fn(batch, outputs, rngs=rngs)

# Generation
generated_audio = model.generate(
    n_samples=1,
    max_length=16000,  # 1 second at 16kHz
    temperature=0.9,
    rngs=rngs
)

    Evaluation Metrics¤

    Density-Estimation Metrics¤

    • Cross-entropy loss / NLL — the training objective. Reported per-token for text, per-pixel for images.
    • Perplexity \(\mathrm{PPL} = \exp(\mathrm{NLL})\) — the standard text metric.
    • Bits per dimension (BPD) is total NLL in nats divided by \(D\log2\). If NLL is already a mean over dimensions, divide only by \(\log2\).
    • Coding interpretation — an ideal code matched to \(p_\theta\) assigns information content \(-\log_2p_\theta(x)\) bits to an event. Real arithmetic or range coders add finite-precision, termination, blocking, and metadata overhead; report measured compressed size rather than assuming a universal percentage gap. The lossless-coding chapter supplies the Kraft, source-coding, tokenizer, and full-file accounting needed to make this operational.

    Image / Video / Audio AR Metrics¤

    For VAR, MAGVIT-v2, and audio AR (WaveNet, Mamba audio):

    • FID (Heusel et al., 2017) compares the first two moments of chosen feature embeddings under a Gaussian approximation. VAR reports ImageNet 256² FID 1.73; sample count, feature implementation, split, conditioning, and reference statistics belong with the number.
    • CMMD is a kernel distance in a contrastive embedding space. State the encoder, estimator, sample count, and preprocessing.
    • FAD applies a Fréchet-style comparison to chosen audio embeddings. Results depend on embedding model, windowing, sample rate, and reference corpus.
    • None of these metrics alone establishes prompt adherence, temporal coherence, memorization safety, or human preference.

    Inference / Throughput Metrics¤

    • Tokens per second and time to first token — useful serving metrics only when hardware, precision, batch size, prompt/output lengths, speculative setup, and concurrency are reported.
    • Time-to-first-token (TTFT) vs time-per-output-token (TPOT) — separates prefill from decode.
    • Acceptance rate for speculative decoding — fraction of draft tokens accepted by the verifier.

    Production Considerations¤

    Inference Cost¤

    There is no hardware-independent tokens-per-second table. Report a measurement vector instead:

    Table AR.20. Inference Cost
    Quantity Why it matters
    Target and draft model/version Fixes architecture, parameter count, and kernels
    Total and active parameters Separates storage/capacity from sparse per-token compute
    Hardware, device count, interconnect, precision Determines arithmetic, communication, and memory ceilings
    Batch size, concurrency, prompt/output length Separates online latency from throughput-oriented serving
    TTFT, TPOT, end-to-end latency, tokens/s Distinguishes prefill, decode, and aggregate throughput
    KV-cache bytes/token/layer and maximum resident tokens Makes memory scaling explicit
    Proposal length, accepted tokens, verifier calls Explains speculative-decoding gains or regressions
    Sampling and stop policy Prevents faster outputs caused only by shorter or different decoding

    Worked accounting example: speculative throughput is a ratio of complete costs¤

    If an exact speculative run proposes four tokens per target verification but accepts 2.5 draft tokens on average, its gross accepted output per verification is not automatically a 2.5× speedup. If the draft costs 0.3 target-pass equivalents and verification costs 1.0, an idealized compute-only ratio is approximately \(2.5/(1.0+0.3)=1.92\). Synchronization, memory traffic, rejected-tail resampling, and batching reduce or sometimes improve the realized wall-clock ratio. Measure it.

    Quantisation¤

    • BF16 / FP16 are standard deployment formats, but not universally safe. Validate normalization, softmax, reductions, logits, and long-context attention in higher precision when overflow, underflow, or accumulated error appears; quantization and mixed-precision choices are model- and hardware-dependent.
    • INT8 / FP8 can improve throughput or capacity on supporting hardware, but quality loss depends on calibration data, kernel, scaling granularity, outliers, and which tensors remain high precision.
    • GPTQ / AWQ and related weight-only methods enable lower-bit deployment in selected settings. Report perplexity and task regressions for the exact checkpoint and quantizer; no fixed “less than 2%” rule transfers across models.
    • MoE quantisation adds expert-frequency and routing-distribution concerns. Per-expert or grouped calibration is a design option, not an architecture-independent requirement or guarantee.

    Inference Stacks¤

    Representative inference stacks include:

    • vLLM (Kwon et al., 2023) — introduced PagedAttention and continuous-batching-oriented serving.
    • SGLang (Zheng et al., 2024) — provides structured generation and prefix/KV reuse mechanisms.
    • TensorRT-LLM — an NVIDIA-optimized stack with hardware-specific quantization and parallel execution options.
    • llama.cpp and MLX — representative local and device-oriented ecosystems. Suitability depends on model format, hardware, kernels, and operational requirements.

    Common Production Pitfalls¤

    • KV-cache growth at long contexts — GQA, MQA, MLA, sliding windows, recurrence, compression, paging, and eviction trade memory against quality, compatibility, or compute in different ways.
    • Unsupported factual output is not diagnosed by temperature alone. Data, objective, prompt, retrieval, context use, calibration, and verification all matter; low temperature can make a wrong high-probability answer more repeatable.
    • Repetition loops may respond to truncation, repetition penalties, contrastive objectives, or better training, but each intervention changes behavior and can damage legitimate repetition. Evaluate by domain.
    • Speculative-decoding mis-acceptance changes the target distribution. Exactness tests should compare empirical output laws on small categorical models, not merely compare a few strings.

    Ethical / Safety Considerations¤

    • Risk assessment follows deployment and affected people, not the model-family label. Analyze misuse, representation harms, privacy leakage, memorization, automation bias, access controls, incident response, and domain-specific regulation.
    • Statistical watermarking (Kirchenbauer et al., 2023) biases a compatible sampling procedure and can be added without retraining in some deployments. Detectability, text quality, secret-key management, edits, translation, paraphrase, short samples, and incompatible decoding limit any “works on every AR model” claim.
    • Provenance metadata such as C2PA can record signed origin and edit information when the generation and publishing pipeline preserves it. Metadata is not proof that unmarked content is synthetic or that marked content is truthful.

    For the broader cross-family picture and how AR fits alongside diffusion / VAE / GAN / EBM / Flow systems, see The Generative Modeling Landscape.

    Independent practice and assessment map: practice and evaluation¤

    Assume the public constructors shown above exist, but do not assume that a successful constructor call validates generation. Your deliverable is a test and measurement plan that another team could reproduce.

    P1 — Bring an autoregressive service from constructor to evidence¤

    Choose one of the Transformer, PixelCNN, or WaveNet examples and declare its observation representation, target shift, output law, and generation unit.

    1. P1.1 — Predict before computing. Predict which correctness test is most likely to fail if a training loss looks excellent but free-running samples collapse immediately: shape compatibility, suffix invariance, target shift, or throughput. Justify the diagnostic order rather than asserting one universal cause.
    2. P1.2 — Debugging task. Specify executable tests for logit/target alignment, suffix invariance, nucleus-boundary retention, prompt preservation, EOS stopping, and RNG reproducibility. Give the smallest finite input and expected invariant for each test.
    3. P1.3 — Interface audit. Classify every block in this chapter as a live Artifex API example, executable framework reference, or shape-oriented pseudocode. Instantiate the chosen documented class in a minimal local test, record the constructor fields and dummy-input shape, and list which behavioral claims still require execution rather than inspection.
    4. P1.4 — Systems calculation. Given 3.0 accepted draft tokens per verification, draft cost \(0.25\) target-pass equivalents, and verifier cost \(1.0\), calculate the idealized ratio. Then design a table that measures TTFT, TPOT, end-to-end latency, throughput, KV-cache memory, precision, batch, concurrency, and proposal acceptance on named hardware.
    5. P1.5 — Transfer. Move the protocol from text to 8-bit audio. State what changes in the observation law, likelihood unit, stopping convention, perceptual evaluation, and safety analysis, and what remains an autoregressive invariant.

    Chapter 6 synthesis¤

    Correct practice connects three contracts: the probabilistic contract (shift, mask, normalization), the software contract (current array shapes, RNG ownership, NNX containers, and public methods), and the evidence contract (protocol-qualified metrics). A constructor that runs does not validate a sampler; a high tokens/s value without workload details is not reproducible; a favorable benchmark without contamination and uncertainty controls is not a research conclusion.

    Retrieval practice: practice and evaluation
    1. What four causality or decoding invariants should be tested before trusting generate?
    2. Why is “tokens per second on an H100” incomplete?
    3. Which extra details make an FID result traceable?
    4. Does a decoding-time watermark prove provenance?

    Solutions. (1) Correct logit/target shift, suffix-invariance of earlier logits, inclusion of the top-\(p\) boundary token, and correct prompt/EOS behavior. (2) Model, precision, batch/concurrency, lengths, hardware count, sampler, and latency/throughput regime are missing. (3) Dataset/split, resolution, conditioning, sample count, feature implementation, preprocessing, and reference statistics. (4) No. It is a statistical signal with robustness and false-positive/negative limits; signed provenance metadata addresses a different question.
